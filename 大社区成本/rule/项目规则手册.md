## 四、技术优化口径规则

### 4.1 天账单结构（推荐场景，其他场景同理）

| 列 | 字段 | 含义 |
|----|------|------|
| A | dtm | 日期（天粒度） |
| C | 组件 | CPU / GPU / 存储 / 网络 / 其他SAAS / 云盘 |
| D | 类型 | 混部容器 / LAMBDA / L20 / H20 等 |
| E | 规格 | VCPU / MEM / LAMBDA 等 |
| J | 标准地域 | Shanghai / Nanjing / Hangzhou 等（用于判断云归属）|
| M | 单价单位 | `元/核/月`（CPU）/ `元/卡/月`（GPU）|
| N | 单价 | 月单价 |
| O | 天物理用量 | CPU：vCPU 核天；GPU：卡天 |
| P | 标准系数 | CPU 混部折算 ≈0.528；GPU ≈1.107 |
| Q | 天标准用量 | = O × P（CPU 有效；GPU 此列精度丢失，**不可用**）|
| R | 金额 | = N × Q / 当月天数（当日费用）|

> **⚠️ 列格式差异**：推荐场景账单为 18 列（A~R），搜索/社区场景账单为 21 列（A~U）。
> 搜索/社区列索引：组件=r[3], 单价单位=r[15], 单价=r[16], O=r[17], Q=r[19], **R=r[20]（单位为元，不是万元，需 ÷10000）**
> 推荐场景 R 列单位为万元，搜索/社区 R 列单位为元，**取数前须注意区分**。

### 4.2 核心计算公式

> ⚠️ 以下公式适用于**单天账单**（如推荐1.31.xlsx、推荐2.28.xlsx）

```
# CPU 标准核（当天）
CPU当天标准核数 = sum(Q列)
    过滤条件：组件='CPU' AND 单价单位='元/核/月'
    ⚠️ 排除 MEM 行（单价单位='元/GiB/月'，那是内存计费，不算核数）

# CPU 物理核（当天）  
CPU当天物理核数 = sum(O列)  [同上过滤条件]

# GPU 卡数（当天，用 O 列，Q 列精度丢失不可用）
GPU当天卡数 = sum(O列)  [组件='GPU']

# 单价（物理核维度，各产品自有单价 N 列不同）
某产品物理单价 = N列（直接读取，元/物理核/月 或 元/物理卡/月）

# 金额验证公式（重要！）
R = N × O / 30.5
⚠️ 中台统一用固定月均 30.5 天计费，不论当月实际天数（1月31天、2月28天均如此）
```

**标准系数 P 的含义**：物理核 → 标准核的折算系数，反映该产品机型相对标准机型的性能比。
- 混部容器 VCPU：P ≈ 0.528（该代际物理机1核 = 0.528标准核）
- GPU：P ≈ 0.001（GPU Q列数值极小，**不可用**，GPU用量统一用 O 列）
- P 值由中台计费系统写入，**不代表利用率，不代表资源使用效率**

### 5.0 UI 结构命名约定

Dashboard 页面结构分三层，统一使用以下名称（代码注释、文档、沟通均以此为准）：

| 层级 | 名称 | 含义 | 示例 |
|------|------|------|------|
| **场景（Scene）** | 第一层，全局切换 | 业务维度，影响所有模块的数据联动 | 推荐 / 搜索 / 社区 |
| **模块（Module）** | 第二层，横向并列 | 分析维度，每个模块独立成页 | ① 预实对比 / ② 用量分析 / ③ 优化测算 |

#### CPU 归一化单价：标准核 vs 物理核

| 单价维度 | 公式 | 数值（2月截面）| 含义 |
|---------|------|-------------|------|
| **标准核单价** | 总金额 ÷ 标准核数 | **≈ 15.13 元/标准核/月** | 账面均价，不同产品已折算为统一标准核后的加权价 |
| **物理核单价** | 总金额 ÷ 物理核数（即标准核单价 × 超售比）| **≈ 35 元/物理核/月** | 实际硬件单位的摊薄价格；标准核：物理核超售比 ≈ 2.31x |

**超售比**：物理机的1个物理核虚拟出约 2.31 个标准核出售（即超卖）。标准核单价看起来便宜，是因为底层物理核被多个标准核分摊。若同类功能改选超售比低的产品（如 REDKV，物理单价 30 元，接近标准核单价），则真实硬件利用率更高，但标准核单价会上升。

> **Dashboard 实现**（用量分析 Tab2，CPU 产品明细表合计行下方）：
> - 橙黄色行（`bg-amber-50`）：「归一化单价（标准核）」= `OM_DATA[curScene].cpuPrice`
> - 橙色行（`bg-orange-50`）：「归一化单价（物理核）」= `cpuPrice × cpuCores / cpuPhysTot`
> - GPU 仅保留一行（无物理/虚拟区分）：「归一化单价」= `OM_DATA[curScene].gpuPrice`
>
> ⚠️ **单价来源**：Dashboard 直接用 `OM_DATA[curScene].cpuPrice / gpuPrice` 常量（由月天数和截面数据预算好），**不从 cpuTot/cpuCores 实时计算**，避免单位换算出错（历史 Bug：万元/万核 数量级正好对消，但引入中间变量时容易混淆是否需要 ÷ 月天数）。

#### GPU 归一化单价：为什么只有一行

GPU **没有超售机制**：账单中 GPU 的 O 列（物理卡天）和 Q 列（标准卡天）在含义上等同，不存在虚拟化折算。一张 L20 / H20 就对应一张实体卡，因此：

| 单价维度 | CPU | GPU |
|---------|-----|-----|
| 标准核/卡 | ✅ 有（虚拟化折算，P 列标准系数）| ✅ 同物理卡（P ≈ 1） |
| 物理核/卡 | ✅ 有（O 列，与标准核不同）| ✅ 同上，两者相同 |
| 超售比 | ≈ 2.31x（标准核 307 万 ÷ 物理核 133 万）| ≈ 1x（无超售）|
| 归一化单价行数 | **2 行**（标准核 + 物理核）| **1 行**（两者等价，只展示一行）|

> GPU 归一化单价 ≈ 2,537 元/卡/月，代表推荐场景所有 GPU 型号（L20/H20/A10 等）按卡数加权后的均价。型号组合不同（如 H20 占比上升），加权均价会随之变化，可用于监测 price mix 趋势。

**单位换算规则（避免踩坑）**：
- CPU：`万核 × 元/核/月 = 万元`（万约掉，**不需要再 ÷10000**）
- GPU：`卡 × 元/卡/月 ÷ 10000 = 万元`（卡不是万卡，**必须 ÷10000**）
- GPU目标卡数：`预算(万元) × 10000 ÷ 单价(元/卡/月) = 卡`
- CPU目标万核：`预算(万元) ÷ 单价(元/核/月) = 万核`
- 存储：`PiB × 元/PiB/月 ÷ 10000 = 万元`

### 6.5 三个问题的计算逻辑

#### 问题①：按 2 月用量，目标月预估

```
CPU月金额 = CPU标准核数(万核) × CPU加权单价(元/核/月)
          = 307.08 × 15.13 = 4,646 万

GPU月金额 = GPU卡数(卡) × GPU加权单价(元/卡/月) ÷ 10000
          = 6,416 × 2,537 ÷ 10000 = 1,628 万

各子项月金额 = R列日均(万元/天) × 30.5
  存储 = 20.55 × 30.5 = 627 万
  SaaS = 62.61 × 30.5 = 1,910 万
  云盘 = 12.99 × 30.5 = 396 万
  网络 = 7.58 × 30.5  = 231 万

月总预估 = 4,646 + 1,628 + 627 + 1,910 + 396 + 231 = 9,437 万

3月预算 = 8,395 万 → 超预算 1,043 万（+12.4%）
```

各子项预算 = 目标月「其他」分项预算 × (各子项2月金额 / 其他合计)，按2月比例拆分。

#### 问题②：资源量调节器

```
CPU优化后量(万核) = 307.08 × (1 - CPU优化比例%)
GPU优化后量(卡)  = 6,416 × (1 - GPU优化比例%)  → 取整
存储优化后量(PiB) = 538.37 - 输入PiB数

优化后金额：
  CPU = 优化后万核 × 15.13（万元）
  GPU = 优化后卡数 × 2,537 ÷ 10000（万元）
  存储 = 原存储月金额 × (优化后PiB / 538.37)（按容量比例折减）
  SaaS/云盘/网络 = 不优化，固定不变

月总 = CPU + GPU + 存储 + SaaS + 云盘 + 网络
vs预算 = 月总 - 目标月预算
```

滑块范围：CPU/GPU 0%~30%（颗粒度1%），存储 0~200 PiB（颗粒度5 PiB）。

#### 问题③：全年逐月打折模型

```
# 前 x 月超支合计
YTD超支 = sum(已知月实际) - sum(已知月预算)
  1~2月：(8,964 + 8,270) - (7,937 + 8,179) = 1,117 万

# 剩余月份总预算
剩余预算 = sum(3~12月原预算) = 93,103 万

# 每月需打折比例
折扣率 z = YTD超支 / 剩余预算 = 1,117 / 93,103 = 1.20%

# 打折后月允许支出
打折预算_m = 原预算_m × (1 - 1.20%)

# CPU目标量（由打折CPU预算反推）
CPU月预算_adj = CPU分项预算_m × (1 - 1.20%)
CPU目标(万核) = CPU月预算_adj ÷ 15.13（元/核/月）

# GPU目标量（由打折GPU预算反推）
GPU月预算_adj = GPU分项预算_m × (1 - 1.20%)
GPU目标(卡) = GPU月预算_adj × 10000 ÷ 2,537（元/卡/月）

# vs当前
CPU差异 = CPU目标 - 307.08（正=可增加，负=需减少）
GPU差异 = GPU目标 - 6,416
```

**全年汇总行**：原预算合计 109,220，打折后总支出（含1~2月实际）≈ 109,220（超支被后续打折弥平），CPU/GPU 取 12 月目标量。

### 9.1 量价分解（PVM）模块

**位置**：② 用量分析 → CPU 区块 / GPU 区块，各产品明细表下方。

**分析方法**：三因子分解（Price-Volume-Mix），对比基准 = 2月 vs 1月环比。

| 因子 | 公式 | 含义 |
|------|------|------|
| Volume Effect | `(Q2总量 - Q1总量) × P1混合单价` | 总量变化的贡献 |
| Mix Effect | `Q2总量 × (P1各SKU加权 - P1混合单价)` | 结构变化的贡献 |
| Price Effect | `Q2各SKU量 × (P2 - P1)` | 单价变化的贡献 |
| 校验 | `ΔCost = Volume + Mix + Price`（误差应 = 0） | 模型自平衡 |

**口径说明**：
- **CPU**：SKU = 产品类型（LAMBDA/LARC/REDKV…），量 = 物理核（physData），价 = amt ÷ physData（元/物理核/月）
- **GPU**：SKU = 型号（L20/H20…），量 = 卡数（data），价 = amt × 10000 ÷ 卡数（元/卡/月）
- 社区 GPU 全部 price = null（按服务类型计费，无法对应型号），不做量价拆解，只展示成本变动说明

**推荐场景验证结果（2月 vs 1月）**：

| 场景 | ΔCost | Volume | Mix | Price | 校验误差 |
|------|-------|--------|-----|-------|---------|
| CPU | -6.19万 | -2.37万(38%) | -3.65万(59%) | -0.17万(3%) | 0.0000 ✅ |
| GPU | +8.95万 | +8.21万(92%) | +0.74万(8%) | 0.00万(0%) | 0.0000 ✅ |

> CPU 物理核从 253.1→249.3 万核，整体减量（Volume 非主因）；结构向低价产品移动（Mix 是主因，KAFKA等贵产品减量）。GPU L20/H20-141G 卡数增加是量效应主因。

#### 第一区块：Recap 文字 + 全年折线图（左右两列）

替换原来的 6 个 KPI 小卡，改为：

**左卡（Recap文字）**：
- 第一句：1~2月实际合计 / YTD预算 / 超支金额 / YTD预算使用率%
- 第二句：以最近月日均平推全年 → 预计全年支出 → vs全年预算差额
- 底部拆解：CPU / GPU / 其他 / 合计，各自YTD金额 + 超支额（红/绿）

**右卡（全年折线图）** `id="ch-annual-forecast"`：
- 🔵 蓝色实线：已知实际（1~2月）
- 🔵 蓝色虚线：未知月按最近月日均平推
- 🔴 红色虚线：逐月预算
- ⚫ 灰色虚线：FY25同期实际（来自 EFF[sc].hist[i].cost × 1.06）

#### 第二区块：资源量逐月预测（替换原①②）

原"①按月预估 + ②全年逐月模型"表格改为一张宽卡，左右两列：

- **左列**：CPU标准核逐月折线 + 明细表（实际/预估/目标/差值）
- **右列**：GPU标准卡逐月折线 + 明细表（实际/预估/目标/差值）
- 顶部保留 3 个小 KPI：YTD超支 / 剩余预算 / 基准折扣率

> 已知月（蓝色）= 按实际金额×预算比例反推核/卡数；未知月（虚线）= 当前截面量平推；目标量（红线）= 按打折预算反推

#### 第三区块：资源量调节器（不变）
